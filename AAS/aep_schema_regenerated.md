# Architecture Evolution Proposal (AEP) Schema (Regenerated)

## Atrahasis Agent System

Architecture Evolution Proposals are used when inventions require
changes to the system architecture.

------------------------------------------------------------------------

# Purpose

AEPs allow the Atrahasis Agent System to evolve its architecture in a
controlled and traceable manner.

They are generated when:

-   architecture gaps are discovered
-   subsystem redesign is required
-   new architecture layers are proposed
-   protocol or governance changes are necessary

------------------------------------------------------------------------

# AEP Structure

Each Architecture Evolution Proposal must include the following fields.

## Metadata

-   AEP_ID
-   Title
-   Authoring Agent
-   Date
-   Status

------------------------------------------------------------------------

## Problem Statement

Describe the architectural limitation in the current system.

------------------------------------------------------------------------

## Invention Concept

Describe the invention or improvement that triggered the proposal.

------------------------------------------------------------------------

## Architecture Gap

Explain why the existing architecture cannot support the invention.

------------------------------------------------------------------------

## Proposed Architecture Change

Describe the architecture modification required.

Possible changes include:

-   subsystem introduction
-   subsystem redesign
-   architecture layer modification
-   protocol updates

------------------------------------------------------------------------

## Affected Layers

List all architecture layers affected by the change.

Examples:

-   Infrastructure
-   Runtime
-   Registry
-   Identity
-   Protocol
-   Agent
-   Governance
-   Bridge

------------------------------------------------------------------------

## Data Model Changes

Describe modifications to system data models.

------------------------------------------------------------------------

## Protocol Implications

Describe any protocol changes required.

------------------------------------------------------------------------

## Security Implications

Explain impacts on identity, trust, and security models.

------------------------------------------------------------------------

## Governance Implications

Describe changes required in councils or governance structure.

------------------------------------------------------------------------

## Migration Strategy

Explain how the system transitions safely from the current architecture
to the new architecture.

------------------------------------------------------------------------

## Implementation Tasks

List the tasks required to implement the architecture change.

------------------------------------------------------------------------

# Decision Outcomes

Councils may return:

-   Approved
-   Approved with modifications
-   Rejected
-   Deferred

------------------------------------------------------------------------

# Versioning

Each AEP references the Canonical System State Model version against
which it was proposed.

Example:

CSSM Version: AAS-State-v2.14

------------------------------------------------------------------------

# Objective

The AEP framework ensures that architecture evolution remains
structured, traceable, and governed.
