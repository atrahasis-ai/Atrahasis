# Atrahasis AASL Parser Architecture

Version: 1.0.0  
Status: Canonical Subsystem Specification  
Date: 2026-03-08

---

# 1. Purpose

This document defines the canonical architecture for the **AASL Parser**, the subsystem that converts written AASL into a deterministic machine-operable semantic representation.

The parser is the first operational boundary where AASL stops being structured text and becomes a formal internal language substrate for the Atrahasis runtime.

This document exists to fully specify:

- parser role and system position
- parser design principles
- input and output contracts
- lexer architecture
- grammar and parse model
- CST and AST construction
- section handling
- symbol collection and reference tracking
- parser modes
- error handling and recovery
- extensibility model
- performance expectations
- package boundaries
- test strategy
- implementation sequencing

The parser does **not** decide final semantic truth. It is responsible for producing a faithful structured interpretation of source input so that validation, canonicalization, runtime loading, query execution, formatting, and developer tooling can operate against a stable representation.

---

# 2. System Role

The AASL Parser is the semantic ingress layer for the AASL ecosystem.

It sits between raw source material and all downstream formal subsystems.

High-level position in Atrahasis:

External Source or `.aas` Document  
→ File Reader / Source Intake  
→ **AASL Parser**  
→ Validator  
→ Canonicalizer  
→ Runtime Object Loader  
→ Query Engine / Memory / Agents / Tooling

The parser also operates inside the AASC compiler pipeline:

English / Markdown / JSON / Draft Structured Input  
→ AASC semantic extraction  
→ Draft AASL  
→ **AASL Parser**  
→ Validator + Canonicalizer  
→ Canonical AASL Output

Because of this dual role, the parser must support both:

- official canonical documents
- draft/generated AASL produced during compilation
- partially valid authoring states in editor environments

# 3. Architectural Objectives

The parser must satisfy the following system objectives.

## 3.1 Determinism

The same input must always produce the same token stream, parse structure, diagnostics ordering, and semantic output.

## 3.2 Non-probabilistic Behavior

No parse behavior may rely on statistical inference, model interpretation, hidden heuristics, or prompt-time guessing.

## 3.3 Layer Purity

The parser builds structure. The validator proves admissibility. The canonicalizer proves deterministic representation. The runtime executes or loads semantic objects. Responsibilities must not blur.

## 3.4 Loss Awareness

The parser must preserve enough source information for:

- diagnostics
- formatting
- editor tooling
- reverse rendering
- source-level debugging
- provenance of parse decisions

## 3.5 Extensibility

The parser must allow ontology growth, additional object templates, and future language modules without requiring a redesign of the parser core.

## 3.6 Incremental Tooling Support

The parser must support partial re-parse and fault-tolerant structure recovery for IDE/editor workflows.

## 3.7 Canonical Compatibility

The parser output must be suitable for downstream canonicalization and hashing without hidden state or source-only semantics leaking into runtime meaning.

---

# 4. Scope of Responsibility

The parser is responsible for:

- reading normalized AASL source text
- identifying document-level sections
- recognizing object declarations
- tokenizing structural and lexical elements
- building a Concrete Syntax Tree (CST)
- building a Semantic Abstract Syntax Tree (AST)
- collecting source spans and token offsets
- collecting symbol declarations
- collecting outbound references
- emitting parser diagnostics
- providing structured handoff artifacts to downstream systems

The parser is **not** responsible for:

- full ontology membership enforcement
- semantic policy enforcement
- trust scoring
- verification logic
- canonical reordering decisions
- storage admission decisions
- claim truth determination
- federation policy

---

# 5. Supported Inputs

The parser must accept AASL originating from the following source classes:

- `.aas`
- `.aas.md`
- `.aas.txt`
- AASL snippets embedded in larger documents or transport payloads
- AASL generated by the AASC compiler
- partially complete authoring input in an editor

## 5.1 Supported Structural Forms

The parser must support:

### Canonical single-line object form

```aasl
AGT{id:ag.research.01 role:research}
```

### Multi-line object form

```aasl
VRF{
  target:c.correlation.001
  status:verified
  method:consensus
  hash:vh.9921
}
```

### Full document form

```aasl
HEADER{...}
LEXICON{...}
OBJECTS{...}
RELATIONS{...}
PROVENANCE{...}
VERIFICATION{...}
FOOTER{...}
```

### Draft / incomplete object form

```aasl
CLM{id:c.1 type:correlation subject:var.temp
```

In tolerant and editor modes, incomplete forms must not crash the parser.

# 6. Outputs and Handoff Contracts

The parser must emit three primary outputs.

## 6.1 Concrete Syntax Tree (CST)

The CST preserves source-level syntax details.

It must retain, where available:

- exact token order
- whitespace and newline boundaries relevant to tooling
- comments when preservation mode is enabled
- exact brace positions
- exact field order from source
- incomplete and recoverable structures
- source spans and offsets for all retained nodes

Primary consumers:

- formatter
- IDE extension
- syntax highlighting engine
- source-preserving transforms
- reverse renderer

## 6.2 Semantic AST

The AST normalizes source syntax into semantic language-level nodes.

It must represent:

- document node
- section nodes
- object nodes
- field nodes
- typed value nodes
- source span metadata
- parse status metadata
- reference lists

Primary consumers:

- validator
- canonicalizer
- runtime loader
- query engine
- memory ingestion
- visualizers

## 6.3 Diagnostics Stream

The diagnostics stream captures parse failures, warnings, and recoverable defects.

Each diagnostic record must include:

- stable diagnostic code
- severity
- message
- source span
- line/column
- expected token or rule
- actual token or observed structure
- recovery action if recovery was attempted
- mode-specific disposition

Primary consumers:

- CLI
- IDE/editor tooling
- compiler pipeline
- build system
- migration tools

# 7. Parser Package Boundaries

The parser subsystem should be implemented as a dedicated package:

`aasl-parser`

Recommended internal modules:

- `source/` — source intake and offset normalization
- `tokens/` — token definitions and token utilities
- `lexer/` — lexical scanner
- `grammar/` — grammar model and parse routines
- `cst/` — concrete syntax tree node definitions
- `ast/` — semantic AST node definitions
- `symbols/` — symbol and reference collection
- `diagnostics/` — diagnostic codes and payload builders
- `modes/` — strict, tolerant, editor, canonical mode policies
- `incremental/` — partial parse and region invalidation support
- `api/` — parser entrypoints and public contracts

The parser package must not depend on runtime services, storage services, or agent logic.

Allowed downstream dependencies should be one-way only:

`aasl-parser` → consumed by validator, canonicalizer, compiler, runtime, tooling

Not allowed:

validator → parser internals mutation  
runtime → parser side effects  
storage → parser-only semantics injection

# 8. Source Intake Architecture

Before lexical analysis, the parser must perform source intake normalization.

## 8.1 Intake Responsibilities

- accept UTF-8 text input
- normalize line endings to a single internal representation
- preserve original byte offsets where required
- preserve character offsets and line/column mapping
- assign parse request identity
- attach optional file metadata
- support embedded-snippet parsing with base offset mapping

## 8.2 Source Object Contract

Each parse request should produce a source envelope containing:

- source id
- source kind
- source path or transport metadata if available
- normalized content buffer
- original content hash if available
- line index table
- byte offset map if needed by storage/provenance systems

This source envelope is immutable for the lifetime of the parse.

---

# 9. Lexical Architecture

The lexer is the first active subcomponent in the parser pipeline.

Its purpose is to convert raw characters into a deterministic stream of typed tokens.

## 9.1 Lexical Design Principles

- single-pass where possible
- deterministic token classification
- minimal contextual dependence
- stable token spans
- no semantic inference
- support for comment preservation
- support for editor fault tolerance

## 9.2 Core Token Classes

The lexer must emit at least the following token classes.

### Structural tokens

- `{`
- `}`
- `:`
- newline
- EOF

### Section tokens

- `HEADER`
- `LEXICON`
- `OBJECTS`
- `RELATIONS`
- `PROVENANCE`
- `VERIFICATION`
- `FOOTER`

### Object type tokens

- `AGT`
- `MOD`
- `DS`
- `TSK`
- `ACT`
- `CLM`
- `EVD`
- `CNF`
- `PRV`
- `VRF`
- `CST`
- `TIM`

### Lexical value classes

- identifier
- string
- number
- boolean
- bare symbol/value token
- comment
- whitespace token when CST-preserving mode is enabled

### Key token

A field key token must be recognized for forms such as:

- `id`
- `role`
- `target`
- `status`
- `subject`
- `method`

## 9.3 Token Metadata

Every token must carry:

- token kind
- raw lexeme
- normalized lexeme where applicable
- byte start / byte end
- char start / char end
- line / column start
- line / column end
- trivia attachment or neighboring trivia references if used

## 9.4 Lexical Error Handling

The lexer must emit explicit token diagnostics for:

- unterminated strings
- illegal characters
- malformed numbers
- impossible token boundaries
- invalid control characters in restricted contexts

The lexer may emit placeholder/error tokens in tolerant and editor modes.

# 10. Grammar Model

The AASL grammar is intentionally constrained and should remain simple enough to support deterministic hand-written parsing.

## 10.1 High-Level Grammar

### Document

A document consists of:

- optional comments and trivia
- zero or more sections
- optionally top-level objects where allowed by mode or migration support
- terminal EOF

### Section

A section consists of:

- section name token
- opening brace
- zero or more entries
- closing brace

### Object

An object consists of:

- object type token
- opening brace
- zero or more field-value pairs
- closing brace

### Field

A field consists of:

- key token
- colon token
- value node

### Value

A value may be:

- identifier
- string
- number
- boolean
- future structured literal if later approved by spec

## 10.2 Grammar Boundaries

The parser must distinguish clearly between:

- document structure errors
- object declaration errors
- field syntax errors
- tokenization errors

The grammar layer must never silently upgrade malformed syntax into valid semantics.

## 10.3 Formal Grammar Ownership

The grammar should be represented in both:

- human-readable narrative form in the language specification
- machine-oriented parser rule definitions in code and tests

This dual representation protects against undocumented parser drift.

# 11. Parser Pipeline Stages

The parser should operate as a staged pipeline.

## 11.1 Stage 1 — Source Intake

Input normalization and source envelope construction.

Output:

- immutable source envelope

## 11.2 Stage 2 — Lexical Analysis

Scan source and emit token stream.

Output:

- ordered token stream
- lexical diagnostics

## 11.3 Stage 3 — Concrete Parsing

Build source-faithful syntax structure from token stream.

Responsibilities:

- identify sections
- identify objects
- parse field/value sequences
- preserve malformed structure where recoverable
- preserve comments/trivia when configured

Output:

- CST
- structural diagnostics

## 11.4 Stage 4 — Semantic Construction

Convert CST structures into normalized AST nodes.

Responsibilities:

- normalize object nodes
- classify field values
- associate objects to sections
- attach parse status and spans
- generate semantic node ids or internal handles

Output:

- semantic AST

## 11.5 Stage 5 — Symbol and Reference Collection

Walk AST to collect declarations and references.

Responsibilities:

- build symbol table
- collect outbound references
- detect duplicate declarations at parse scope level
- record unresolved references without deciding final semantic validity

Output:

- symbol table
- reference table
- duplicate declaration diagnostics

## 11.6 Stage 6 — Diagnostic Consolidation

Merge lexical, structural, and collection diagnostics into a stable stream.

Output:

- ordered diagnostics bundle

## 11.7 Stage 7 — Handoff Packaging

Assemble a parser result contract for downstream consumers.

Output:

- CST handle
- AST handle
- symbols
- references
- diagnostics
- parse mode metadata
- parser version metadata

# 12. Concrete Syntax Tree Design

The CST must preserve exact source structure closely enough to support authoring tools and source-preserving transforms.

## 12.1 CST Node Families

Recommended CST node families:

- `DocumentCstNode`
- `SectionCstNode`
- `ObjectCstNode`
- `FieldCstNode`
- `ValueCstNode`
- `CommentCstNode`
- `TriviaCstNode`
- `ErrorCstNode`

## 12.2 CST Preservation Requirements

The CST should retain:

- original field ordering
- source comments
- whitespace blocks if formatting tools require them
- incomplete object bodies
- unexpected tokens attached to nearest recoverable node
- brace positions and colon positions

## 12.3 CST Stability Rule

The CST is not the execution structure and must not be mutated by downstream semantic layers.

Any normalized or canonical representation must be derived from the AST or later formal structures, not by editing CST nodes in place.

---

# 13. Semantic AST Design

The AST is the parser’s principal semantic output.

It must normalize syntax into language-level nodes while preserving traceability back to source.

## 13.1 Core AST Node Families

Recommended node families:

- `DocumentNode`
- `SectionNode`
- `ObjectNode`
- `FieldNode`
- `IdentifierValueNode`
- `StringValueNode`
- `NumberValueNode`
- `BooleanValueNode`
- `UnknownValueNode`
- `ErrorValueNode`

## 13.2 Object Node Contract

Each object node should include at minimum:

- internal node id
- AASL object type
- parsed `id` field if present
- section membership
- field map
- field order as parsed
- source span
- parse status
- outbound references
- raw-to-normalized value record if needed

## 13.3 Parse Status Model

Every AST node should have parse status metadata, such as:

- `complete`
- `partial`
- `recovered`
- `invalid`

This allows tolerant and editor mode consumers to reason over incomplete structure without pretending it is fully valid.

## 13.4 AST Purity Rule

The AST may normalize syntax but must not invent missing semantics.

Examples:

Allowed:

- classify `verified` as an identifier or enum-like bare value
- normalize number token form
- preserve source span on normalized field node

Not allowed:

- infer missing `type` from neighboring objects
- guess omitted braces
- invent absent references
- auto-attach ontology meaning not present in source or parser rule tables

---

# 14. Section-Aware Parsing

The parser must understand document sections as first-class structural entities.

## 14.1 Canonical Sections

The parser must recognize these sections:

- `HEADER`
- `LEXICON`
- `OBJECTS`
- `RELATIONS`
- `PROVENANCE`
- `VERIFICATION`
- `FOOTER`

## 14.2 Section Semantics at Parse Layer

At parse layer, section handling is structural, not fully semantic.

The parser must:

- detect section boundaries
- attach child objects to section containers
- preserve section order from source
- identify duplicate sections when prohibited by grammar rules
- retain enough structure for canonical order checking later

## 14.3 Why Section Awareness Matters

Section-aware parsing is required for:

- canonical document re-emission
- validator enforcement
- file writer correctness
- bundle construction
- storage indexing
- document inspection tools

---

# 15. Symbol Table and Reference Collection

AASL is reference-based, so the parser must collect symbol declarations and outbound references as explicit artifacts.

## 15.1 Symbol Table

The symbol table records declared identifiers discovered during parse.

Recommended entry fields:

- symbol id
- object type
- node handle
- section
- source span
- declaration status

Example declarations:

- `ag.research.01` → `AGT`
- `ds.climate.44` → `DS`
- `c.correlation.001` → `CLM`

## 15.2 Reference Table

The reference table records outbound references found in fields that point to other objects.

Recommended entry fields:

- source node handle
- source object id if present
- field name
- referenced id
- source span
- resolution status

## 15.3 Parse-Time Resolution

The parser may attempt lightweight resolution against in-document declarations, but unresolved references are not always fatal in the parse layer.

Permitted parse outcomes:

- resolved
- unresolved but structurally valid
- malformed reference token
- duplicate declaration collision

Full semantic validity remains a validator responsibility.

---

# 16. Comments and Non-Semantic Material

AASL comments begin with `#`.

The parser must support two policies.

## 16.1 Ignore Policy

Comments are discarded from semantic structures and ignored by runtime/validation consumers.

## 16.2 Preservation Policy

Comments are preserved in CST and possibly in source maps for:

- IDE tooling
- formatting
- documentation generation
- reverse rendering
- author assistance

Comments must never alter semantic meaning unless a future formal extension explicitly defines a semantic comment form.

---

# 17. Parser Modes

The parser must support multiple operating modes because Atrahasis uses AASL in production, tooling, migration, and editor environments.

## 17.1 Strict Mode

Used for:

- production document compilation
- CI checks
- trusted build pipelines
- memory admission preparation

Behavior:

- fails on structural invalidity
- fails on malformed tokens that prevent safe interpretation
- emits hard diagnostics
- does not suppress fatal parse defects

## 17.2 Tolerant Mode

Used for:

- migration pipelines
- draft conversion workflows
- exploratory compiler stages
- document repair tooling

Behavior:

- attempts structure recovery
- emits diagnostics
- produces partial AST when possible
- separates recoverable defects from hard blockers

## 17.3 Editor Mode

Used for:

- IDE extensions
- live authoring
- autocomplete and inline diagnostics

Behavior:

- incremental parsing support
- very high fault tolerance
- preserves incomplete structures
- prefers continuity of user feedback over hard abort

## 17.4 Canonical Parse Mode

Used when:

- a document is being prepared for validation and canonicalization
- canonical re-emission and hashing are about to occur

Behavior:

- expects normalized structure
- minimizes recovery behavior
- produces AST optimized for downstream canonical processing

---

# 18. Error Handling and Recovery

Parser diagnostics must be precise, stable, and actionable.

## 18.1 Diagnostic Categories

The parser must at minimum distinguish:

- lexical errors
- structural syntax errors
- section-structure errors
- object declaration errors
- malformed field-value pair errors
- duplicate declaration errors at parse scope
- recovery notices

## 18.2 Common Structural Errors

Examples:

- missing closing brace
- missing opening brace
- missing colon
- invalid object declaration
- unexpected token in field position
- object nested in invalid context

## 18.3 Common Lexical Errors

Examples:

- malformed identifier
- unterminated string
- invalid number literal
- illegal character

## 18.4 Recovery Strategy

Recovery should be rule-driven and mode-aware.

Permitted strategies include:

- skip to next newline
- skip to next closing brace
- skip to next likely object type token
- skip to next section token
- create `ErrorCstNode` or partial AST node and continue

The parser must record recovery actions explicitly so downstream consumers know the parse was repaired rather than clean.

## 18.5 No Silent Semantic Repair

The parser must never silently transform invalid semantics into valid meaning.

Examples of forbidden behavior:

- inventing a missing object id
- filling absent target references from context
- inferring a field key from neighboring tokens without an explicit recovery rule

---

# 19. Extensibility Model

The parser must be future-compatible with AASL growth.

## 19.1 Stable Core Grammar, Extensible Vocabulary

The parser core should keep grammar stable while allowing expansion in:

- ontology terms
- object templates
- value constraints enforced later by validation
- field-level admissibility tables

## 19.2 Object-Type Growth

New object types should be addable through governed registration tables and parser token/grammar extensions without rewriting the parser architecture.

## 19.3 Section Growth

Future sections may be introduced only through formal specification change. The parser must reject unknown sections in strict/canonical modes unless an approved extension registry says otherwise.

## 19.4 Plugin Boundary

Parser extensibility should be declarative, not arbitrary code injection into the parse core.

Preferred mechanism:

- versioned grammar registry
- section registry
- object type registry
- reserved token tables

This preserves determinism and reviewability.

---

# 20. Interaction with Downstream Systems

## 20.1 Validator

The parser hands off:

- AST nodes
- symbol table
- reference table
- diagnostics
- source spans

The validator then determines:

- template conformity
- ontology membership
- required field presence
- semantic type correctness
- reference legitimacy
- admissibility outcomes

## 20.2 Canonicalizer

The parser provides the structured basis for canonical ordering and deterministic re-emission.

The canonicalizer must not have to reinterpret raw text.

## 20.3 Runtime

The runtime loader consumes validated AST or parser-derived structures to instantiate live semantic objects.

## 20.4 Query Engine

The query engine depends on stable object and field representation that the parser establishes.

## 20.5 Tooling

Formatters, editors, visualizers, and explorers depend on CST/AST dual output and precise source spans.

---

# 21. Incremental Parsing for Tooling

Because authoring environments need low-latency feedback, the parser should support incremental re-parse.

## 21.1 Incremental Requirements

- region invalidation by span
- token cache reuse where safe
- section-local reparse where possible
- stable node identities where safe for editor diffing
- low-latency diagnostics updates

## 21.2 Editor Integrity Rule

Incremental parsing must never produce a semantic result that could not be produced by a full parse of the same current text, apart from transient partial-node metadata.

---

# 22. Performance Expectations

The parser must be designed for predictable latency and low overhead.

## 22.1 Performance Goals

- linear or near-linear behavior on ordinary documents
- bounded recovery cost in malformed input
- no catastrophic backtracking
- memory usage proportional to document size and preservation mode

## 22.2 Performance Strategy

- hand-written deterministic parser preferred over complex generalized parser frameworks
- avoid hidden regex-heavy ambiguity paths for full grammar handling
- separate CST preservation mode from lean semantic mode where optimization is required

## 22.3 Large Document Handling

The parser should support large semantic bundles without requiring full editor-preservation data when running in production strict/canonical modes.

---

# 23. Security and Safety Considerations

The parser is part of the trusted ingress path and must be hardened.

## 23.1 Required Properties

- no code execution during parse
- no dynamic evaluation of values
- no schema downloading during parse
- no network calls during parse
- bounded resource consumption
- explicit handling of malformed or hostile input

## 23.2 Adversarial Input Considerations

The parser must be tested against:

- brace storms
- pathological nesting attempts
- oversized identifiers
- comment flooding
- malformed unicode / control character injection
- repeated duplicate declarations
- truncated transport payloads

The parser package should expose a small stable public API.

Recommended entrypoints:

- `parseDocument(source, mode)`
- `parseObject(source, mode)`
- `parseSection(source, mode)`
- `tokenize(source, options)`
- `reparseRegion(existingState, changeSet)`

Recommended result contract:

- parser version
- mode
- source metadata
- token stream or token handle
- CST or CST handle
- AST or AST handle
- symbol table
- reference table
- diagnostics
- success / partial / failed status

# 25. Canonical Internal Data Contracts

The parser should standardize the following internal contracts.

## 25.1 ParseResult

Contains the complete parse handoff bundle.

## 25.2 Token

Stable lexical record with precise spans.

## 25.3 Diagnostic

Stable diagnostic payload with machine-readable code.

## 25.4 SymbolEntry

Represents a declaration discovered during parse.

## 25.5 ReferenceEntry

Represents an outbound reference discovered during parse.

## 25.6 ParseModePolicy

Encodes mode-specific behavior for recovery, fatality thresholds, and preservation.

These contracts must be versioned and shared downstream through explicit package interfaces rather than ad hoc object shapes.

# 28. Non-Negotiable Rules

The following parser rules are mandatory.

1. The parser must be deterministic.  
2. The parser must not introduce hidden semantics.  
3. The parser must preserve source traceability.  
4. The parser must support strict and tooling-oriented modes.  
5. The parser must separate structure-building from semantic admissibility.  
6. The parser must expose symbols, references, and diagnostics explicitly.  
7. The parser must be extensible through governed registries, not ad hoc hacks.  
8. The parser must be usable by validator, canonicalizer, runtime, compiler, and tooling without reinterpretation of raw text.

# 32. Final Statement

The AASL Parser is not merely a reader for `.aas` files.

It is the formal structural boundary that allows Atrahasis to move from text to deterministic semantic computation.

If the parser is weak, every downstream subsystem becomes unstable or forced to compensate with hidden logic. If the parser is explicit, deterministic, section-aware, and contract-driven, then validator, canonicalizer, runtime, compiler, query, storage, and tooling can all operate on a common language substrate.

That is why the parser is the correct first standalone subsystem specification in the AASL documentation set.
