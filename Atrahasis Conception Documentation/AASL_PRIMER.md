## 1. What This Primer Is

This primer teaches how to think in AASL before trying to master every normative detail.

AASL, the Atrahasis Agent Specification Language, is the canonical declarative language used to describe agents, tasks, tools, workflows, memory resources, policies, messages, execution context, and semantic relationships in a form that is both human-readable and machine-stable.

This document is explanatory. It is meant to help a person or model become productive quickly.

This document does **not** replace the normative specification. When any example, phrasing, or simplification in this primer differs from the normative rules, the normative rules in `AASL_SPECIFICATION.md` and the canonical architecture documents control.

---

## 2. Why AASL Exists

Most agent systems fail at the same boundary.

They can generate text, plans, JSON fragments, and tool calls, but they cannot reliably turn those outputs into durable system objects that survive across time, environments, toolchains, governance regimes, and multi-agent exchanges.

AASL exists to solve that problem.

It gives Atrahasis a single declarative object language that can be used to:

- describe what exists in the system,
- express how objects relate to one another,
- attach constraints and policy,
- preserve identity over time,
- compile source into a canonical semantic graph,
- validate admissibility before runtime,
- support query, storage, federation, and audit.

In practical terms, AASL is the language that turns “agentic output” into “governed system reality.”

---

## 3. How To Think About AASL

Do not think of AASL as “just another config language.”

Do not think of it as code that gets executed line by line.

Do not think of it as a glorified JSON wrapper.

Think of AASL as a **semantic declaration language**.

When you write AASL, you are declaring a world of typed objects and stable relationships.

That world is then:

1. parsed into structure,
2. validated against rules,
3. compiled into canonical semantics,
4. admitted into storage and runtime systems,
5. consumed by agents, runtimes, queries, and governance layers.

AASL answers questions like these:

- What agent exists?
- What role does it hold?
- What tools can it access?
- What task is it authorized to execute?
- What workflow does that task belong to?
- What policy constrains the execution?
- What memory resources may be read or written?
- What identities and references remain stable over time?
- What should happen if an implementation sees an unknown extension?

AASL is therefore closer to a typed semantic graph language than to imperative code.

---

## 4. Core Mental Model

A useful mental model is:

**AASL source text -> structure -> meaning -> canonical graph -> runtime objects**

### 4.1 Source Layer
This is the `.aas` file written by a human or generated by a compiler.

### 4.2 Structural Layer
The parser turns source text into tokens, blocks, declarations, and abstract syntax structures.

### 4.3 Semantic Layer
The compiler resolves types, references, namespaces, relations, and constraints into explicit meaning.

### 4.4 Canonical Layer
The system normalizes the document into a deterministic semantic representation so the same document means the same thing everywhere.

### 4.5 Runtime Layer
The runtime resolves executable bindings, policies, memory links, and operational affordances.

### 4.6 Governance Layer
Registries, ontology versions, compatibility rules, and admission controls determine what is allowed into the system and how it evolves.

## 8. First Example: Read It Conceptually

Below is a **teaching example**, not a grammar contract. The exact concrete syntax of an implementation may evolve with the normative parser reference. The point here is to teach the shape of the language.

```aas
meta {
  id: "doc://demo/customer-support"
  version: "1.0.0"
}

namespace demo.customer_support

agent SupportCoordinator {
  id: "agent://support/coordinator"
  role: @CoordinatorRole
  tools: [@TicketLookup, @KnowledgeBaseSearch]
  policies: [@PIIHandlingPolicy]
}

role CoordinatorRole {
  id: "role://support/coordinator"
  capabilities: [@TicketTriage, @EscalationRouting]
}

tool TicketLookup {
  id: "tool://support/ticket-lookup"
  kind: "api"
}

policy PIIHandlingPolicy {
  id: "policy://security/pii-handling"
  effect: "restrict"
}
```

Conceptually, this document says:

- the document has identity and version metadata,
- the declarations live in a namespace,
- there is an agent named `SupportCoordinator`,
- that agent has a stable identifier,
- the agent refers to a role object,
- the agent refers to tool objects,
- the agent is constrained by policy,
- those other referenced objects are also declared in the document.

This is enough to understand the heart of AASL: you declare a graph of governed objects.

## 9. The Most Important Ideas To Learn Early

### 9.1 Objects Matter More Than Lines
AASL is object-centered, not line-centered.

A single declaration may span many lines, but what matters semantically is the object it defines and the relationships that object participates in.

### 9.2 Identity Is Not Decoration
Identifiers are not optional fluff. They are what let the same conceptual thing be tracked across storage, queries, compilers, migrations, and federation boundaries.

### 9.3 References Are Semantic Links
When one declaration points to another, it is not merely string substitution. It establishes a semantic relationship that must be resolvable and meaningful under the active ontology and resolver policy.

### 9.4 Namespaces Prevent Chaos
Namespaces keep extension, evolution, and reuse from collapsing into ambiguity.

### 9.5 Validation Happens Before Runtime Trust
AASL documents are not supposed to become reality merely because they parse. They must also validate semantically and pass admission policy.

### 9.6 Runtime Is A Consumer, Not The Source Of Meaning
The runtime may act on an object, but it does not invent the object’s meaning. Meaning originates from the language, ontology, validation, and compilation pipeline.

## 10. Reading AASL Step By Step

When you open an unfamiliar AASL document, read it in this order.

### Step 1: Identify the document scope
Look for metadata, namespace, and version.

Ask:

- What domain is this document about?
- What namespace does it live in?
- What version claims are attached?

### Step 2: Find the major object declarations
Scan for the object types being declared.

Ask:

- Is this mostly agents?
- Mostly tasks and workflows?
- Mostly policy and constraints?
- Mostly memory or data resources?

### Step 3: Identify stable IDs
Find what identifiers are being used.

Ask:

- Which objects have durable IDs?
- Which identifiers are local symbols only?
- Which identities are intended for cross-document use?

### Step 4: Follow references
Find how objects point to one another.

Ask:

- What role does the agent point to?
- Which task points to which workflow?
- Which policy constrains which object?
- Which memory store is attached to which runtime?

### Step 5: Look for constraints and policies
Not every object relationship is permissive.

Ask:

- What is allowed?
- What is required?
- What is forbidden?
- What admission or validation conditions exist?

### Step 6: Infer the object graph
At this point, you should be able to mentally sketch the graph:

- nodes are declarations,
- edges are references and typed relations,
- policies and constraints restrict allowable operational behavior.

---

## 11. Writing AASL Step By Step

When you author AASL, use this workflow.

### Step 1: Decide what objects actually exist
Do not start with syntax. Start with the ontology.

Ask:

- What types of things am I representing?
- What are the stable entities here?
- What are transient values versus durable objects?

### Step 2: Decide object boundaries
A common mistake is collapsing many distinct concepts into one giant declaration.

Separate objects when they have:

- different lifecycles,
- different identities,
- different policies,
- different provenance,
- different relationships,
- different reuse needs.

### Step 3: Assign stable identities where durability matters
If an object will be referenced later, queried independently, audited, versioned, federated, or updated across time, it likely needs stable identity.

### Step 4: Put the document in the correct namespace
Do not author serious documents in an ambiguous or generic namespace. Namespace is part of semantic hygiene.

### Step 5: Declare objects with explicit relationships
Be clear about how one object points to another. Hidden assumptions create ambiguity.

### Step 6: Attach policy and constraint where it belongs
Do not bury security or governance assumptions in comments. Express them as real declarations and real semantic relationships.

### Step 7: Validate and canonicalize
A document is not finished when it “looks right.” It is finished when it validates and compiles to the intended canonical meaning.

## 12. AASL Versus JSON, YAML, and Code

### 12.1 Versus JSON
JSON is a general data format.

AASL is a semantic declaration language with identity, validation classes, namespaces, relation semantics, ontology evolution, compilation, and runtime portability expectations.

### 12.2 Versus YAML
YAML is friendly for human editing, but it does not by itself solve semantic determinism, canonical identity, or governed extensibility.

### 12.3 Versus Imperative Code
Code usually describes *how* to execute behavior.

AASL primarily describes *what semantic objects exist* and *how they relate*, leaving execution to runtimes and bound operational systems.

### 12.4 Versus Ad Hoc Prompt Output
Prompt output is often unstable, underspecified, and difficult to govern.

AASL is specifically designed to transform expressive system knowledge into something durable and machine-trustworthy.

## 13. Understanding Declarations

A declaration is the basic authoring act in AASL.

A declaration usually does four things:

1. identifies the kind of thing being declared,
2. gives that thing a local name or symbol,
3. assigns properties,
4. connects it to other things.

Teaching form:

```aas
<Type> <LocalName> {
  id: "<stable-identifier>"
  property: <value>
  relation: @OtherObject
}
```

Example:

```aas
capability TicketTriage {
  id: "capability://support/ticket-triage"
  domain: "customer_support"
}
```

Conceptually, this means: “There exists a capability object called `TicketTriage` with this stable identity and this assigned domain property.”

---

## 14. Understanding Properties

Properties attach structured meaning to an object.

Examples of property shapes include:

- scalar values,
- strings,
- numbers,
- booleans,
- lists,
- maps,
- references,
- nested blocks,
- constrained expressions supported by the active ontology.

Example:

```aas
memory_store IncidentMemory {
  id: "memory://ops/incidents"
  retention_days: 90
  encrypted: true
}
```

This declaration assigns ordinary semantic facts to the object.

The important point is that properties are not merely arbitrary fields. Their meaning depends on the type system and ontology.

---

## 15. Understanding References

References are one of the most important parts of AASL.

A reference says that one declared object points to another object in a semantically meaningful way.

Teaching example:

```aas
task EscalateComplexCase {
  id: "task://support/escalate-complex-case"
  assigned_role: @CoordinatorRole
}
```

This should be read as:

“The `EscalateComplexCase` task is semantically linked to the `CoordinatorRole` object.”

References may be:

- local,
- qualified,
- cross-document,
- registry-resolved,
- namespace-sensitive,
- identity-backed.

Good references are resolvable, explicit, and stable.

Bad references are vague, accidental, or dependent on hidden context.

---

## 16. Understanding Namespaces

Namespaces are how AASL keeps a growing ecosystem coherent.

Without namespaces, extension eventually becomes collision.

With namespaces, different teams and modules can define new types, relations, and patterns without breaking each other.

Teaching example:

```aas
namespace atrahasis.support.v1
```

This tells the reader and the system that the declarations in the document live under a specific semantic domain.

A namespace often implies:

- ownership,
- ontology scope,
- version context,
- compatibility expectations,
- resolution behavior.

When in doubt, use a specific namespace and keep imports explicit.

---

## 17. Understanding Identity

Identity is one of the pillars of AASL.

A local symbol like `SupportCoordinator` is useful for reading a document, but a stable identifier is what lets the object remain itself across time and across systems.

Why identity matters:

- query needs it,
- storage needs it,
- federation needs it,
- audit needs it,
- updates need it,
- provenance needs it,
- canonicalization depends on it.

Good identity practice:

- assign durable IDs to durable objects,
- use stable schemes consistently,
- avoid accidental reuse,
- avoid meaning-changing identifier drift.

Poor identity practice:

- changing IDs casually,
- using display labels as IDs,
- relying only on local names,
- mixing temporary handles with canonical identity.

---

## 18. Understanding Relations

Some semantics are better represented as explicit relations rather than merely embedded properties.

Example thinking:

- an agent **uses** a tool,
- a task **belongs_to** a workflow,
- a policy **constrains** a runtime,
- a memory node **derives_from** a source,
- a capability **enables** an action,
- an event **emitted_by** an agent.

The exact relation syntax is controlled by the active grammar and ontology, but the mental model is stable: relations create typed edges in the graph.

When designing a model, ask whether a connection is best represented as:

- a literal property,
- a reference property,
- a first-class relation,
- or a higher-order object with its own identity and provenance.

---

## 19. Understanding Constraints and Policies

AASL is not only about what exists. It is also about what is permitted, required, restricted, or inadmissible.

A policy may describe:

- access restrictions,
- execution restrictions,
- privacy handling,
- memory retention,
- federation trust rules,
- validation requirements,
- publication rules.

A constraint may describe:

- cardinality,
- type requirements,
- admissible values,
- relation requirements,
- required provenance,
- version compatibility,
- state transition guards.

Do not reduce policy to comments or external tribal knowledge. In AASL systems, policy should be attached semantically where it can be validated and governed.

---

## 20. Understanding Validation

AASL validation is not one thing.

A document can fail at multiple layers.

Typical validation categories include:

- lexical validity,
- syntactic validity,
- structural validity,
- type validity,
- reference validity,
- ontology validity,
- canonicalization validity,
- policy admissibility,
- portability or profile compatibility.

A key lesson for new authors:

A document that parses is not necessarily valid.

A document that validates is not necessarily admissible.

A document that is admissible is not necessarily executable in every runtime profile.

This layered thinking is essential.

---

## 21. Understanding Compilation

Compilation in AASL does not merely mean “turn text into machine code.”

It means turning source declarations into explicit, canonical semantic objects.

The compiler typically does work such as:

- resolving imports,
- binding namespaces,
- resolving references,
- expanding or normalizing declarations,
- assigning canonical internal forms,
- extracting semantic graph edges,
- generating intermediate representations,
- emitting canonicalized output suitable for storage and query.

A good way to think about AASL compilation is:

**source meaning made explicit**

---

## 22. Understanding Canonicalization

Canonicalization matters because Atrahasis requires semantic determinism.

Two authors may write semantically equivalent documents in slightly different surface forms. The system still needs one stable canonical meaning.

Canonicalization may normalize things like:

- ordering,
- identifier representation,
- resolved references,
- default values,
- collection forms,
- relation edge forms,
- structural sugar,
- provenance scaffolding.

For new authors, the lesson is simple:

Write clearly, then let the canonicalization pipeline produce the normalized representation.

Do not depend on accidental formatting or incidental source order for meaning unless the specification says order is meaningful.

## 23. A Minimal Teaching Walkthrough

Below is a slightly larger teaching example.

```aas
meta {
  id: "doc://ops/incident-response"
  version: "1.0.0"
}

namespace atrahasis.ops.incident_response

workflow IncidentResponse {
  id: "workflow://ops/incident-response"
  stages: [@Detect, @Triage, @Resolve]
}

role IncidentCommander {
  id: "role://ops/incident-commander"
  capabilities: [@OpenIncident, @AssignResponder, @CloseIncident]
}

agent CommanderAgent {
  id: "agent://ops/commander"
  role: @IncidentCommander
  workflows: [@IncidentResponse]
  policies: [@RestrictedDataHandling]
}

task Detect {
  id: "task://ops/detect"
}

task Triage {
  id: "task://ops/triage"
}

task Resolve {
  id: "task://ops/resolve"
}

policy RestrictedDataHandling {
  id: "policy://security/restricted-data"
  effect: "restrict"
}
```

How to read it:

- there is a workflow called `IncidentResponse`,
- the workflow refers to three task objects,
- there is a role called `IncidentCommander`,
- the role has capabilities,
- there is an agent called `CommanderAgent`,
- the agent is linked to the role,
- the agent is associated with the workflow,
- the agent is also constrained by policy.

The semantic graph is now richer than a flat config file. It expresses governed relationships.

## 24. Common Authoring Patterns

### 24.1 The Identity-First Pattern
Use when the object is durable and likely to be referenced elsewhere.

- define the object,
- assign a stable ID,
- reference it from dependent declarations.

### 24.2 The Role-Capability-Agent Pattern
Use when modeling execution responsibility.

- define capabilities,
- group them into a role,
- assign the role to an agent,
- constrain the agent with policy.

### 24.3 The Task-Workflow Pattern
Use when organizing operational sequences.

- define tasks independently,
- define the workflow as the object that references and orders them,
- avoid collapsing all task semantics directly into one monolithic workflow body.

### 24.4 The Policy Attachment Pattern
Use when semantics are allowed or constrained by governance.

- define a policy object,
- attach it explicitly to the governed object,
- avoid hidden policy assumptions.

### 24.5 The Resource Binding Pattern
Use when an agent, task, or workflow depends on memory stores, queues, runtimes, or external resources.

- define the resource as a first-class object,
- attach identity and provenance,
- bind consuming objects to that resource by reference.

### 24.6 The Namespace Module Pattern
Use when a domain has many related declarations.

- put them in a coherent namespace,
- use explicit imports,
- version the module thoughtfully.

## 25. Common Mistakes

### Mistake 1: Treating AASL like arbitrary nested data
AASL is not just a place to dump fields.

Fix: model stable semantic objects and explicit relations.

### Mistake 2: Omitting stable identity for durable objects
If an object matters across time, it usually needs a durable ID.

Fix: define an identity scheme early.

### Mistake 3: Hiding relationships inside strings
A string that happens to mention another object is not the same thing as a typed reference.

Fix: use real references and typed relations.

### Mistake 4: Using comments as policy
Comments may help readers, but they do not participate in validation and governance.

Fix: declare policies and constraints explicitly.

### Mistake 5: Blurring separate objects together
One declaration should not contain multiple lifecycles unless the ontology truly requires it.

Fix: split objects cleanly.

### Mistake 6: Ignoring namespaces
Flat naming does not scale.

Fix: use disciplined namespace ownership.

### Mistake 7: Assuming parse success means system readiness
Parsing is only the first gate.

Fix: validate, canonicalize, and test query/runtime assumptions.

### Mistake 8: Depending on implementation quirks
AASL is meant to be portable.

Fix: author against the specification and approved profiles, not incidental parser behavior.

## 26. Guidance For AI Agents Writing AASL

If you are an AI system generating AASL, follow these rules.

### 26.1 Model the ontology before emitting syntax
First determine what kinds of objects exist, then generate declarations.

### 26.2 Prefer explicitness over clever compression
Do not hide meaning in shorthand if that increases ambiguity.

### 26.3 Emit stable references only when resolvable
Do not invent cross-document references without a grounding basis.

### 26.4 Separate facts from guesses
If identity, namespace, or ontology membership is uncertain, mark the uncertainty through the approved authoring or compilation path instead of hallucinating certainty.

### 26.5 Avoid fake semantics
Do not fabricate types, relations, policy fields, or resolver syntax that are not defined by the active specification or ontology profile.

### 26.6 Preserve portability
Do not rely on one tool’s undocumented extensions unless the target profile explicitly allows them.

### 26.7 Prefer canonical patterns
When multiple source shapes could express the same idea, choose the most canonical and clearest one.

## 27. Guidance For Human Authors

### 27.1 Start with a semantic sketch
Before writing source, sketch:

- the object types,
- the stable identities,
- the key references,
- the constraints,
- the namespace.

### 27.2 Write small, valid documents first
Do not begin with a massive cross-module graph if you are still learning the language.

### 27.3 Keep object boundaries clean
Good AASL often comes from good modeling more than fancy syntax.

### 27.4 Let tooling help you
Use formatter, validator, compiler preview, and query explorer tooling rather than guessing.

### 27.5 Think long-term
Ask whether the document will still be readable, valid, and governable months later when other teams and other runtimes consume it.

## 29. A Quick Mental Checklist Before You Save

Before considering an AASL document “done,” ask:

- Does every durable object have stable identity?
- Are the right objects separated cleanly?
- Are all important relationships explicit?
- Is the namespace correct and specific?
- Are policies and constraints attached semantically?
- Are references resolvable?
- Does the document validate?
- Will the canonicalized meaning match my intent?
- Could another team or another agent read this without hidden assumptions?

If the answer to several of these is no, the document is not finished.

## 30. Quick Anti-Patterns

### Anti-pattern: giant catch-all object
Everything is stuffed into one declaration.

Why it is bad: identity, lifecycle, policy, and reuse become muddy.

### Anti-pattern: stringly-typed references
Relationships are implied in strings.

Why it is bad: the compiler cannot reason safely about them.

### Anti-pattern: namespace neglect
The document has generic naming and no clear ownership.

Why it is bad: collision and ambiguity accumulate over time.

### Anti-pattern: comment-driven governance
Security and policy live only in prose comments.

Why it is bad: validators and runtimes cannot enforce comments.

### Anti-pattern: non-deterministic authoring assumptions
The author assumes field order, parser quirks, or local environment state will always remain the same.

Why it is bad: portability breaks.

Why this is useful:

- one object references the other,
