# PART 1 — DESIGN ACTIONS

## DA-01: Per-Modality Monthly Snapshot Extraction

### Snapshot Definition

A **Monthly Behavioral Snapshot (MBS)** for agent a at month t and modality m is the aggregated feature vector computed from all Behavioral VTDs generated by agent a during calendar month t.

### Extraction Method

For each modality m in {T, S, E, R, L}:

1. Collect all Behavioral VTDs for agent a from calendar month t
2. Compute the per-feature mean across all VTDs in the month
3. z-score normalize each feature against the population statistics for that month (same normalization as C17 Section 4)
4. Concatenate into modality snapshot vector MBS_m(a, t)

### Snapshot Dimensionality

| Modality | Features Aggregated | Snapshot Dims | Notes |
|----------|-------------------|--------------|-------|
| Temporal (T) | T-01 to T-07 means | 24 | Same as C17 BFE temporal vector |
| Structural (S) | S-01 to S-06 means | 32 | Same as C17 BFE structural vector |
| Error (E) | E-01 to E-05 means | 23 | Same as C17 BFE error vector |
| Resource (R) | R-01 to R-04 means | 21 | Same as C17 BFE resource vector |
| Lexical (L) | L-01 to L-05 means | 753 | Same as C17 BFE lexical vector |

### Minimum VTDs per Month

A monthly snapshot is valid only if agent a has >= 5 Behavioral VTDs in that calendar month. Months with fewer VTDs are marked as missing. Trajectory comparison requires >= 6 valid monthly snapshots out of any 9 consecutive months (allowing up to 3 gaps).

### Storage

Monthly snapshots are stored alongside Behavioral VTDs in AiSIA-controlled storage. Retention: 24 months (longer than raw VTD retention of 12 months, because trajectory comparison needs longer history).

## DA-02: Population-Mean Drift Computation and Subtraction

### Population-Mean Trajectory

For each modality m and each calendar month t, compute the population-mean snapshot:

```
MBS_pop_m(t) = TrimmedMean({ MBS_m(a, t) : a in active_agents }, trim=0.10)

Where:
  TrimmedMean removes the top and bottom 10% of values per dimension
  before computing the mean. This resists bias from same-origin clusters.
  active_agents = all agents with valid MBS for month t
```

### Idiosyncratic Snapshot

```
MBS_idio_m(a, t) = MBS_m(a, t) - MBS_pop_m(t)
```

The idiosyncratic snapshot captures agent-specific behavioral features after removing population-level systematic drift.

### Computation Cadence

- Population-mean snapshots are recomputed monthly (at month end)
- Historical population means are fixed once computed (not retroactively adjusted)
- This ensures trajectory comparisons are deterministic — the same data always produces the same result

### Phase 0-1 Consideration

During Phase 0-1 (< 1,000 agents), the population mean is unstable. Mitigation: require a minimum population of 50 agents with valid MBS before activating population-mean de-trending. Below this threshold, use raw MBS without de-trending and apply a higher trajectory similarity threshold to compensate for systematic drift confounds.

## DA-03: DTW Configuration

### Step Pattern

**Symmetric2** (Sakoe-Chiba classification Type IV):
- Allowed steps: (1,0), (0,1), (1,1)
- Weights: 1, 1, 2 respectively (diagonal steps count double to normalize for path length)
- This prevents pathological warpings where one time point maps to many

### Sakoe-Chiba Band Width

**w = 2 months.** The warping path cannot deviate more than 2 months from the diagonal. This means:
- Agent A's month 3 can align with Agent B's months 1-5
- Prevents pathological warpings (e.g., mapping 6 months of A to 1 month of B)
- Justified by the assumption that same-origin agents are updated within a ~2-month window of each other

If FEASIBILITY monitoring flag MF-3 indicates staggered updates > 2 months are common, the band width can be increased to 3 via governance (AiSIA + Stiftung board).

### Point Distance Function

For each modality m, the point distance between two monthly snapshots is:

```
point_dist_m(MBS_idio_m(a_i, t_s), MBS_idio_m(a_j, t_r)) =
  1 - cosine_similarity(MBS_idio_m(a_i, t_s), MBS_idio_m(a_j, t_r))
```

Cosine distance is used uniformly across modalities (rather than C17's modality-specific metrics) because trajectory comparison operates on aggregate monthly snapshots rather than raw VTD distributions. The distributional structure that justifies JSD/Wasserstein for point-in-time comparison is already aggregated into the snapshot vectors.

### DTW Normalization

```
DTW_raw_m(a_i, a_j) = DTW distance from dynamic programming (sum of point distances along optimal path)
DTW_norm_m(a_i, a_j) = DTW_raw_m / path_length

Where path_length = number of steps in the optimal warping path.

DTW_norm_m in [0, 1] by construction (cosine distance per point is in [0, 1]).
```

Path-length normalization ensures that longer trajectories (more months) do not automatically produce higher distances.

### Handling Missing Months

If either agent has a missing month within the trajectory window:
- The missing month is linearly interpolated from adjacent valid months
- Maximum consecutive interpolated months: 2. If more than 2 consecutive months are missing, the trajectory is split into segments, each compared separately, and the segment distances are averaged.

## DA-04: Drift Direction Vector Specification

### Cumulative Displacement Vector

For each modality m and agent a with valid snapshots at months t_1, t_2, ..., t_n:

```
drift_dir_m(a) = MBS_idio_m(a, t_n) - MBS_idio_m(a, t_1)
```

The cumulative displacement vector captures the total direction of drift from the agent's earliest available snapshot to its most recent.

### Direction Distance

```
d_dir_m(a_i, a_j) = 1 - |cosine_similarity(drift_dir_m(a_i), drift_dir_m(a_j))|
```

Absolute cosine similarity is used per FEASIBILITY Attack 9 — both correlated (cosine ~ +1) and anti-correlated (cosine ~ -1) drift indicate a shared cause. Independent agents should have uncorrelated drift (cosine ~ 0).

**Range:** d_dir_m in [0, 1]. d_dir_m = 0 when drift directions are perfectly (anti-)correlated. d_dir_m = 1 when drift directions are orthogonal (uncorrelated).

### Drift Magnitude Anomaly Detection

Compute drift magnitude for each agent:

```
mag_m(a) = ||drift_dir_m(a)||_2
```

If mag_m(a) < 0.1 * population_mean_magnitude_m for 3+ consecutive months, the agent is flagged for **drift suppression anomaly** (FEASIBILITY Attack 4 defense). Agents with near-zero idiosyncratic drift are added to the drift-suppression watch list.

## DA-05: Direction-Shape Fusion Formula

### Per-Modality Trajectory Distance

For each modality m:

```
d_Traj_m(a_i, a_j) = beta * d_dir_m(a_i, a_j) + (1 - beta) * DTW_norm_m(a_i, a_j)

Where:
  beta = 0.60 (drift direction contributes 60%)
  (1 - beta) = 0.40 (DTW shape contributes 40%)
```

**Rationale for beta = 0.60:** Drift direction is the primary signal (IDEATION core insight: agents from the same creator drift in similar directions). DTW shape is supplementary (captures timing and pattern of drift). The 60/40 split reflects this priority while still utilizing both signals.

### Cross-Modality Aggregation

```
d_Traj(a_i, a_j) = Sum_{m in {T,S,E,R,L}} (w_m_traj * d_Traj_m(a_i, a_j))

Where w_m_traj uses C17's adversary weights (renormalized to sum to 1.0):
  w_T_traj = 0.30
  w_S_traj = 0.25
  w_E_traj = 0.20
  w_R_traj = 0.15
  w_L_traj = 0.10
```

The trajectory modality inherits the same adversary-weighted modality ordering as C17's point-in-time comparison: temporal features are hardest to fake in trajectory space (hardware and architecture determine long-term timing patterns), lexical features are easiest.

## DA-06: Multi-Task Trajectory Consistency (rho_Traj)

### Per-Task-Category Trajectories

Instead of one trajectory per modality, compute trajectories **per SEB task category** (7 categories from C17 Section 7.2). For each modality m and task category c:

```
MBS_m_c(a, t) = mean of BFE features from VTDs in category c during month t
```

This produces up to 35 trajectory pairs per agent pair (5 modalities x 7 categories).

### Trajectory Consistency Coefficient

```
For each modality m, compute d_Traj_m across each task category c independently:
  d_Traj_m_c(a_i, a_j) for c in {logical, math, creative, code, factual, analysis, adversarial}

rho_Traj_m = 1 - CV(d_Traj_m_c1, d_Traj_m_c2, ..., d_Traj_m_c7)

Where CV = coefficient of variation (std / mean), clamped to [0, 1].
```

**Interpretation:** If trajectory similarity in modality m is consistent across task categories (low CV, high rho_Traj_m), it indicates shared origin rather than coincidence. If similarity is high in one category but low in others (high CV, low rho_Traj_m), it is likely coincidental.

### Integration with Trajectory Distance

The final trajectory distance for the B formula:

```
d_Traj_final(a_i, a_j) = d_Traj(a_i, a_j) * rho_Traj_aggregate

Where:
  rho_Traj_aggregate = mean(rho_Traj_m) across modalities m with valid trajectories
```

Note: d_Traj is a *distance* (lower = more similar). The trajectory *similarity contribution* in the B formula is:

```
trajectory_contribution = w_Traj * (1 - d_Traj_final) * rho_Traj_aggregate

Wait — this double-counts rho. Correction:

B formula contribution for trajectory modality:
  w_Traj * d_Traj(a_i, a_j) * rho_Traj_aggregate

This is subtracted in the B formula:
  B = 1 - Sum(w_m * d_m_mean * rho_m) - w_Traj * d_Traj * rho_Traj_aggregate
```

Actually, to maintain consistency with C17's formula structure where rho gates the contribution:

```
trajectory_dissimilarity_term = w_Traj * d_Traj(a_i, a_j) * rho_Traj_aggregate
```

This term is added to the other modality dissimilarity terms and subtracted from 1 in the B formula. See DA-07 for the complete updated formula.

## DA-07: Updated 6-Modality Weight Allocation

### Updated B Formula (Phase 2, Trajectory Active)

```
B(a_i, a_j) = 1 - [ Sum_{m in {T,S,E,R,L}} (w'_m * d_m_mean * rho_m)
                     + w_Traj * d_Traj * rho_Traj ]

Where:
  d_Traj = d_Traj(a_i, a_j) from DA-05 (already aggregated across modalities)
  rho_Traj = rho_Traj_aggregate from DA-06
  w'_m = trajectory-adjusted weights (below)
  w_Traj = trajectory weight
```

### Weight Allocation Table

| Modality | C17 Weight (5-mod) | C19 Weight (6-mod, trajectory active) | Change |
|----------|-------------------|---------------------------------------|--------|
| Temporal (T) | 0.30 | 0.26 | -0.04 |
| Structural (S) | 0.25 | 0.22 | -0.03 |
| Error (E) | 0.20 | 0.17 | -0.03 |
| Resource (R) | 0.15 | 0.13 | -0.02 |
| Lexical (L) | 0.10 | 0.08 | -0.02 |
| **Trajectory (Traj)** | **N/A** | **0.14** | **+0.14** |
| **Total** | **1.00** | **1.00** | |

**Derivation:** Each modality donates weight proportionally to its current weight. Temporal donates the most (0.04 from 0.30), lexical donates the least (0.02 from 0.10). This preserves the adversary-weighted ordering while making room for the trajectory modality.

**Note:** These match C17 Section 11.4 exactly. C19 validates the placeholder weights.

### Weight Constraints

All C17 constraints carry forward:
- w_m in [0.05, 0.40] for all m (including w_Traj)
- Sum(w_m) + w_Traj = 1.00
- No modality below 0.05 (lexical at 0.08 is above minimum)
- Weight changes require AiSIA proposal + Stiftung board ratification + 30-day notice

### Activation/Deactivation

- **Trajectory active:** Both agents have >= 6 valid monthly snapshots with >= 6-month overlap. Use 6-modality weights.
- **Trajectory inactive:** One or both agents lack sufficient history. Use C17's standard 5-modality weights (no trajectory term). No reweighting — the 5-modality weights remain at their C17 values (0.30, 0.25, 0.20, 0.15, 0.10).

## DA-08: Discontinuity Detection and Trajectory Reset Protocol

### Discontinuity Definition

A discontinuity in agent a's trajectory at month t for modality m is detected when:

```
||MBS_m(a, t) - MBS_m(a, t-1)|| > mu_m + 3 * sigma_m

Where:
  mu_m = population mean of ||MBS_m(*, t) - MBS_m(*, t-1)|| (mean monthly change magnitude)
  sigma_m = population std of monthly change magnitude
```

A change exceeding 3 standard deviations from the population mean indicates a structural break (model swap, major architecture change, new training data).

### Discontinuity Handling

1. **Confirm:** If discontinuity detected in >= 3 of 5 modalities simultaneously, classify as a **model architecture change event**. Single-modality discontinuities are treated as noise.
2. **Reset:** Agent's trajectory history is partitioned into pre-discontinuity and post-discontinuity segments. Only the post-discontinuity segment is used for future trajectory comparison.
3. **Cool-down:** Post-reset, the agent needs 6 new monthly snapshots before trajectory comparison reactivates. During cool-down, 5-modality weights apply.
4. **Counter:** Track the number of trajectory resets per agent per year. If resets > 1 per year, the agent is flagged for **frequent reset anomaly** and added to enhanced monitoring.

### Governance

- Discontinuity threshold (3 sigma) is adjustable by AiSIA (range: [2, 5] sigma)
- Reset frequency limit (1/year) is adjustable by AiSIA (range: [1, 3] per year)

# PART 2 — PRE-MORTEM

## What Could Go Wrong?

**F-01: Population-mean de-trending removes too much signal.** If the agent population is homogeneous (most agents use the same 2-3 model families), the population mean captures most of the meaningful drift, leaving only noise in the idiosyncratic component.

*Mitigation:* Monitor the ratio of idiosyncratic drift magnitude to total drift magnitude. If idiosyncratic drift is < 20% of total drift for > 50% of agents, the de-trending is too aggressive. Fallback: reduce trim percentage or switch to model-family-specific de-trending (subtract mean within model family, not across population).

**F-02: DTW band width too narrow.** If agents are commonly updated with 3+ month stagger, the 2-month band misses alignments.

*Mitigation:* Monitor the distribution of optimal DTW path deviations. If > 20% of same-origin pairs (from training data) have paths that hit the band boundary, increase band width. Governed parameter, adjustable by AiSIA.

**F-03: Per-task-category trajectories have insufficient data.** Some SEB categories have only 3-5 tasks. Monthly VTDs in a specific category may be sparse, producing noisy per-category trajectories.

*Mitigation:* Require minimum 3 VTDs per task category per month for a per-category trajectory to be valid. If fewer categories have valid trajectories, rho_Traj is computed over available categories (minimum 3 of 7).

# PART 3 — SIMPLIFICATION PASS

**Can anything be removed without losing the core value?**

1. **Per-task-category trajectories (DA-06)?** These add multi-task consistency gating, which is valuable but complex. Without them, trajectory comparison could be gamed by creating coincidental trajectory similarity in one task category. **Keep** — the anti-coincidence gating is essential for a modality with w=0.14.

2. **DTW shape comparison?** Could use drift direction alone (C19-B). This simplifies computation and eliminates DTW configuration. **Evaluate:** If drift direction alone achieves similar detection rates to the hybrid, remove DTW. Decision: **keep for now**, defer to Phase 2 empirical evaluation. The specification should include a governance mechanism to disable DTW if it adds < 5% detection improvement over direction-only.

3. **Lexical trajectory?** Lexical features are the noisiest modality (w_L = 0.10, w_L_traj = 0.08). Lexical drift may be dominated by system prompt changes rather than model updates. **Consider removing** lexical from trajectory comparison. Decision: **keep but monitor**. If lexical trajectory contribution to detection is < 2% over 4 quarters, AiSIA may disable lexical trajectory via governance.
