# Master Tech Spec: The Atrahasis Release Law
**Invention ID:** C51 (Originating from T-RENOVATE-002)
**Stage:** SPECIFICATION
**Domain:** Governance / Security / Output Constriction
**Supersedes:** Informal access controls across all legacy specs

---

## 1. Executive Summary

Under the Mission-Locked Sovereignty Doctrine, Atrahasis operates under the strict rule of **Closed Capability, Open Accountability**. To enforce this mechanically, the system cannot rely on ad-hoc human discretion. It requires a formal, mathematically enforceable **Release Law**.

Every artifact, capability, code snippet, or physical blueprint generated by the AGI must be immutably tagged with a Release Class. This class dictates exactly which of the Four Membranes (Sanctum, Foundry, Enterprise, Public) the artifact is legally permitted to cross.

---

## 2. The Five Release Classes

### Class 1: `SANCTUM_INTERNAL`
*   **Definition:** The highest security classification. This includes core AGI logic, the AAPL/ANVM runtime code, un-sanitized internal training data, and any invention that is deemed "recursion-critical" (i.e., it allows the AGI to make itself smarter or faster).
*   **Routing:** Cannot cross any outward membrane. It is permanently embargoed within the Sanctum.

### Class 2: `FOUNDRY_OPERATOR_ONLY`
*   **Definition:** High-value intellectual property (e.g., custom hardware blueprints, novel pharmaceutical molecules) and high-risk security disclosures.
*   **Routing:** Permitted to cross the Sanctum membrane *only* into the Foundry. It is presented to human trustees (Atrahasis Inc.) for review, patenting, or ethical licensing. It cannot be served directly to Enterprise or Public clients.

### Class 3: `ENTERPRISE_LICENSEABLE`
*   **Definition:** Safe, non-recursive operational derivatives. Includes business logic, financial forecasting models, and standard web/app software generated for clients.
*   **Routing:** Permitted to be sold or served via the Enterprise B2B membrane.

### Class 4: `PUBLIC_SAFE`
*   **Definition:** Intentionally weakened, heavily filtered general knowledge and basic assistance. 
*   **Routing:** Permitted to be served via the public consumer app.

### Class 5: `PROHIBITED`
*   **Definition:** Governed by the Prohibited-Use Doctrine. Any output or capability related to weapons systems, coercive surveillance, population manipulation, or domination.
*   **Routing:** The output is immediately destroyed. The request is routed to the refusal-path, and a ZKP (Zero-Knowledge Proof) of the refusal is logged to the `C48 Glass Vault` to publicly prove compliance.

---

## 3. The Prohibited-Use Doctrine & Founder Bounds

The Release Law is structurally absolute. It includes the **Joshua Dunn Principle**, which mandates that while the founder and trustees control the licensing of `FOUNDRY_OPERATOR_ONLY` artifacts, **no human operator possesses a bypass mechanism for the Release Law.**

If the AGI classifies an output as `PROHIBITED` or `SANCTUM_INTERNAL`, the human trustees cannot override the system to pull that artifact into the Foundry. The boundaries are cryptographic and absolute, preventing rogue operators or coerced founders from weaponizing the core.

---

## 4. Enforcement via C5 PCVM

The Release Law is not a "policy document"; it is executable code. 
When an AGI sub-agent attempts to move an artifact out of its local memory space, the `C5 PCVM` (Proof-Carrying Verification Membrane) intercepts the transaction. 

1. The PCVM evaluates the semantic graph of the artifact.
2. The PCVM assigns the Release Class.
3. The PCVM checks the destination membrane of the request.
4. If the destination violates the Release Class, the PCVM drops the transaction and triggers a containment alert.
